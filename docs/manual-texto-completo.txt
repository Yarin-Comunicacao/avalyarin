1. Introdução e Visão Geral
O Avalyarin é uma plataforma digital desenvolvida para avaliar bares, restaurantes e
estabelecimentos gastronômicos, com foco inicial na região de Pinheiros e Vila Madalena, em
São Paulo. O sistema permite que usuários comuns, influenciadores gastronômicos e críticos
profissionais avaliem suas experiências de forma estruturada, gerando dados confiáveis tanto
para outros consumidores quanto para os próprios donos de estabelecimentos.
O nome “Avalyarin” é uma fusão de “Avaliar” + “Yarin” (nome da agência criadora do projeto). O
sistema foi concebido como um ecossistema completo que conecta cinco públicos distintos:
consumidores que buscam recomendações confiáveis, influenciadores que compartilham
suas experiências gastronômicas, críticos gastronômicos que oferecem análises profissionais,
donos de estabelecimentos que desejam monitorar e melhorar a percepção do público sobre
seus negócios, e agentes de suporte que garantem a qualidade dos dados e atendem
solicitações.
A plataforma opera como um Progressive Web App (PWA), o que significa que funciona
diretamente no navegador do celular como se fosse um aplicativo nativo — pode ser instalado
na tela inicial do telefone, funciona offline em funcionalidades básicas, e envia notificações
push. Não é necessário baixar pela App Store ou Google Play.
Estado atual do sistema (Junho 2026):
187 estabelecimentos ativos com cardápio completo
20 categorias gastronômicas
7 roles de usuário (user, influencer, critic, business, support, admin, owner)
406 testes automatizados passando
16 fases de desenvolvimento concluídas

2. Objetivo do Projeto
O Avalyarin foi criado com os seguintes objetivos estratégicos:
Para o consumidor final: Oferecer um sistema de avaliação mais justo e detalhado do que
plataformas genéricas como Google Maps ou TripAdvisor. No Avalyarin, cada item consumido é
avaliado individualmente, e a nota final é calculada com pesos dinâmicos que consideram
critérios como atendimento, ambiente, custo-benefício e qualidade dos pratos. O sistema de
insígnias e níveis gamifica a experiência, incentivando avaliações frequentes e detalhadas.
Para o influenciador gastronômico: Criar uma plataforma onde suas avaliações tenham peso
diferenciado e visibilidade, permitindo que construam uma audiência fiel e estabeleçam

parcerias comerciais com estabelecimentos. Influenciadores têm avaliações ilimitadas e só
podem avaliar presencialmente (via QR code).
Para o crítico gastronômico: Oferecer um espaço profissional onde avaliações de críticos
verificados recebem um selo de autenticidade (estrela azul safira) e aparecem com prioridade
na lista de avaliações do estabelecimento quando a nota é igual ou superior a 9.
Para o dono de estabelecimento: Fornecer um painel de gestão com insights sobre a
percepção dos clientes, notificações em tempo real sobre novas avaliações, ferramentas de
marketing como códigos promocionais, posts no feed do app, e lista de transmissão para
seguidores.
Para a empresa (Yarin Agência): Monetizar através de planos de assinatura (tanto para
usuários quanto para estabelecimentos), parcerias com influenciadores, e dados agregados
sobre o mercado gastronômico.

3. Arquitetura do Sistema
A arquitetura do Avalyarin segue o modelo cliente-servidor, onde o “cliente” é o que o usuário
vê (a interface visual no celular/computador) e o “servidor” é o computador remoto que processa
as informações e armazena os dados.

3.1 Diagrama de Arquitetura

O diagrama acima ilustra as quatro camadas principais do sistema: o Cliente (PWA) construído
com React 19, Tailwind CSS 4 e tRPC Client; o Servidor com Express 4, tRPC 11, Drizzle ORM,
Sharp para processamento de imagens e LLM API para inteligência artificial; o Armazenamento
com TiDB/MySQL (30+ tabelas) e Amazon S3 (fotos e documentos); e as Integrações com
Manus OAuth, Google Tag Manager, Connect Yarin e Whisper para transcrição.

3.2 Frontend (Interface do Usuário)
O frontend é tudo aquilo que o usuário vê e interage — botões, telas, formulários, animações.
No Avalyarin, o frontend é construído com:
React 19 — a biblioteca de interface mais popular do mundo, criada pelo Facebook (Meta)
em 2013. React permite construir interfaces divididas em “componentes” reutilizáveis (como
peças de LEGO). Cada botão, card de estabelecimento, ou formulário é um componente
independente que pode ser reaproveitado em diferentes telas.
TypeScript 5.9 — uma extensão do JavaScript (a linguagem nativa dos navegadores) que
adiciona “tipos” ao código. Isso significa que, antes mesmo de executar o programa, o
computador verifica se os dados estão no formato correto.

Tailwind CSS 4 — um framework de estilização que permite definir cores, tamanhos,
espaçamentos e layouts diretamente no código HTML, sem precisar criar arquivos CSS
separados.
Wouter — um roteador leve para React que controla qual página é exibida baseado na
URL.
Framer Motion — biblioteca de animações que cria transições suaves entre telas e
elementos.
shadcn/ui + Radix UI — componentes acessíveis e consistentes para formulários, modais,
dropdowns e outros elementos de interface.

3.3 Backend (Servidor)
O backend é o “cérebro” invisível do sistema — processa requisições, valida dados, calcula
notas, e se comunica com o banco de dados:
Node.js 22 — ambiente que permite executar JavaScript fora do navegador, no servidor.
Especialmente eficiente para aplicações em tempo real porque processa múltiplas
requisições simultaneamente.
Express 4 — o framework web que recebe as requisições HTTP e direciona cada uma para
a função correta.
tRPC 11 — tecnologia que conecta o frontend ao backend de forma “type-safe” (segura em
tipos). Se o backend espera receber um userId do tipo número, o frontend é obrigado a
enviar um número — qualquer erro é detectado automaticamente antes de ir ao ar.
Drizzle ORM 0.44 — “tradutor” entre o código TypeScript e o banco de dados MySQL. Em
vez de escrever comandos SQL manualmente, o desenvolvedor escreve código TypeScript
legível e o Drizzle traduz para SQL automaticamente.
Sharp 0.34 — biblioteca de alta performance para processamento de imagens. Converte
automaticamente para WebP, redimensiona e gera thumbnails.

3.4 Banco de Dados
MySQL (hospedado via TiDB Cloud) — MySQL é o sistema de banco de dados relacional
mais popular do mundo. O TiDB é uma versão distribuída e escalável, 100% compatível
com MySQL, que permite crescer horizontalmente adicionando mais servidores.

3.5 Armazenamento de Arquivos
Amazon S3 (Simple Storage Service) — serviço de armazenamento de arquivos da AWS.
Todas as fotos de avaliações, logos de estabelecimentos, documentos de verificação de
idade, e imagens de cardápio ficam guardadas no S3 com 99,999999999% de durabilidade.

5.3 Regras de Governança de Dados
Campos obrigatórios para visibilidade: Um estabelecimento só aparece no app para os
usuários quando possui TODOS os seguintes campos preenchidos: nome, endereço completo
(por extenso), bairro, telefone, Instagram, horário de funcionamento, e cardápio com pelo menos
1 item ( hasMenu = true ). Estabelecimentos que não atendem esses critérios ficam com
status = 'pending' e são invisíveis para usuários comuns.
Padronização de endereços: Todos os endereços devem iniciar com logradouro por extenso
— “Rua”, “Avenida”, “Alameda”, “Travessa”, “Praça”, “Largo”, “Viela”, “Beco”, “Estrada” ou
“Rodovia”. Abreviações como “R.”, “Av.” ou “Al.” são rejeitadas pela validação do backend.
Complemento obrigatório para multi-inquilinos: Quando o endereço contém palavras-chave
como “Shopping”, “Galeria”, “Food Hall”, “Mercado Municipal”, “Food Park”, “Centro Comercial”
ou “Mall”, o campo complement passa a ser obrigatório (mínimo 2 caracteres).
Timestamps em UTC: Todas as datas e horas são armazenadas em formato UTC como Unix
timestamps em milissegundos. A conversão para o fuso horário local (Brasília, GMT-3) é feita
apenas na exibição.
Identificadores visuais (codes): Além do id interno, cada entidade possui um code legível:
Usuários: numérico (1-200000000)
Categorias: ca001 a ca999
Estabelecimentos: es000001 a es999999
Avaliações: ra000001 a ra999999
Grupos: gr000001 a gr999999
Itens do cardápio: mi000001 a mi999999

7. Fluxos do Aplicativo
7.1 Fluxo de Primeiro Acesso (Onboarding)

O diagrama acima detalha o fluxo completo de primeiro acesso ao Avalyarin. As etapas são:
Etapa 1 — Verificação de idade: O app exibe uma tela com checkbox “Tenho mais de 18
anos”, flanqueada por imagens de restaurante diurno e bar noturno. Se o usuário for menor, é
necessário enviar documento (RG/CPF) para aprovação do admin.
Etapa 2 — Tela de boas-vindas: Apresenta o Avalyarin com duas opções: “Cadastre-se” (novo
usuário → login → onboarding) e “Já Tenho Cadastro” (login → home diretamente).

Etapa 3 — Autenticação OAuth: O login é feito via Manus OAuth. O usuário faz login com
email e o sistema cria automaticamente uma conta no Avalyarin vinculada ao seu perfil.
Etapa 4 — Pesquisa de onboarding (7 etapas): Na primeira vez, o app pergunta: data de
nascimento (roleta dia/mês/ano), região de SP, média de consumo, categorias preferidas,
prioridades de avaliação, frequência de visitas, e fontes de descoberta.
Etapa 5 — Captação de geolocalização: O app solicita permissão de localização e salva as
coordenadas no perfil do usuário para a funcionalidade “Perto de Mim”.
Etapa 6 — Home do app: O usuário é direcionado para a Home com seções “Minhas
Preferidas” (5 categorias da survey), “Explore outros grupos” (4 grupos com imagens), e “Veja
todas as Categorias”.

7.2 Fluxo de Avaliação

O fluxo de avaliação é o core do sistema. Para avaliar, o usuário precisa atender aos critérios
de qualificação: ter 18+ anos, nome e sobrenome preenchidos, e username definido. O
sistema oferece dois modos de avaliação:
Modo Direto: nota de 0 a 10 para cada item consumido
Modo Analítico: avalia critérios específicos (atendimento, ambiente, apresentação, sabor,
custo-benefício) e o sistema calcula a nota final com pesos dinâmicos
Após a avaliação, o sistema classifica automaticamente o source (origem) baseado no histórico
de QR code scan:

Presencial: QR scan registrado há menos de 8 horas
Híbrido: QR scan registrado entre 8 e 48 horas
Remoto: sem QR scan registrado

7.3 Fluxo de QR Code

O diagrama de sequência acima mostra o fluxo técnico completo do scan de QR code, desde a
abertura da câmera até o registro no banco de dados com validação de geolocalização
(distância ≤200m do estabelecimento).
Selo Verificado: Quando um usuário completa 3 QR scans em 3 estabelecimentos diferentes,
recebe automaticamente o selo de verificado (critério oculto — o usuário não sabe que precisa
de 3 scans).

7.4 Fluxo do Business (Dono de Estabelecimento)
1. O usuário com role business acessa “Meus Locais” no bottomNav
2. Vê seus estabelecimentos vinculados com ID visível (bares são clicáveis)
3. Pode fazer upload de logo (1:1, 500x500 WebP) e foto de capa (1200x800 WebP)
4. Visualiza informações do estab (nome, endereço, telefone, @) em modo read-only
5. Para alterar informações, clica em “Solicitar alteração ao suporte” que abre chat inline

11. Integrações Externas
11.1 Manus OAuth
O Avalyarin utiliza o Manus OAuth como provedor único de autenticação. O fluxo é:
1. Usuário clica em “Entrar” no app

2. É redirecionado ao portal Manus OAuth ( VITE_OAUTH_PORTAL_URL )
3. Faz login com email (Google, Facebook, etc.)
4. Manus OAuth retorna um código de autorização
5. Backend troca o código por um token de sessão
6. Sessão é armazenada em cookie HttpOnly com JWT assinado
Vantagem: O Avalyarin nunca vê a senha do usuário nem o token bruto do provedor.
Limitação: Se o serviço Manus ficar indisponível, nenhum usuário consegue fazer login.

11.2 Google Tag Manager (GTM)
O GTM está integrado para rastreamento de analytics. O container ID é configurado via tabela
integrations no banco de dados, permitindo que o admin altere sem necessidade de
deploy.

Pageviews (todas as páginas)
Avaliações concluídas
QR scans realizados
Cadastros de novos usuários

11.3 Connect Yarin (Link Bio)

11.4 Whisper (Transcrição de Áudio)

12.3 Proteções Específicas
Rate Limiting: Endpoints sensíveis possuem limitação de taxa para prevenir abuso.
Validação de Geolocalização: O scan de QR code valida que o usuário está a no máximo 200
metros do estabelecimento, prevenindo scans fraudulentos.
Verificação de Idade: Menores de 18 anos precisam enviar documento para aprovação manual
do admin. O documento é armazenado no S3 com acesso restrito.
Dados Sensíveis: Emails e dados pessoais são armazenados no banco com acesso restrito
por role. Apenas admin e owner podem ver emails de outros usuários.

12.4 LGPD — Considerações
O sistema atualmente opera com as seguintes medidas de conformidade parcial com a LGPD:
Consentimento implícito via aceite dos termos no cadastro
Dados mínimos coletados (apenas o necessário para funcionamento)
Conexões criptografadas (TLS) em todas as camadas

Acesso restrito por role a dados pessoais
Pendências para conformidade total:
Implementar hash de PII (dados pessoais identificáveis) em tabelas de auditoria
Modal de consentimento LGPD explícito antes do cadastro
Endpoint de “direito ao esquecimento” (deletar todos os dados de um usuário)
Relatório de impacto à proteção de dados

16. Fases Implementadas — Histórico Completo
O desenvolvimento do Avalyarin foi organizado em 16 fases, todas concluídas com sucesso:

Fase 1 — Fundação e Design System
Definição do design “Neon Urbano” com paleta âmbar/dourado sobre fundo escuro, tipografia
Bebas Neue + Inter + JetBrains Mono, e implementação do tema base com Tailwind CSS 4.

Fase 2 — Categorias e Estabelecimentos
Implementação das 20 categorias gastronômicas, grid de navegação, página de categoria com
filtros, e página individual de estabelecimento com cardápio interativo.

Fase 3 — Sistema de Avaliação
Motor de avaliação com modo Direto e Analítico, cálculo de nota com pesos dinâmicos, upload
de fotos, registro de custos, e classificação automática de source (presencial/híbrido/remoto).

Fase 4 — Autenticação e Perfis
Integração com Manus OAuth, sistema de roles, perfis de usuário com username, onboarding
survey de 7 etapas, e age gate com verificação de documento.

Fase 5 — Grupos e Comunidade
Grupos privados com convites, grupo de seguidores de influencer, eventos com RSVP,
compartilhamento de avaliações, e chat em grupo (140 caracteres).

Fase 6 — Business Panel
Painel para donos de estabelecimentos com gestão de estabs, posts efêmeros, códigos
promocionais, insights de avaliações, e notificações automáticas.

Fase 7 — QR Code e Verificação Presencial
Scanner de QR code com câmera, validação de geolocalização (≤200m), registro de scans,
classificação de source, e selo de verificado (3 scans).

Fase 8 — Insígnias e Gamificação
Sistema de 16 níveis com pontos decrescentes por idade, insígnias por
categoria/bairro/estabelecimento, e ranking pessoal com histórico.

Fase 9 — Admin Panel
Painel administrativo completo com gestão de usuários, estabelecimentos, aprovações de
claims, analytics, gestão de equipe, e configurações do sistema.

Fase 10 — Influencer e Critic
Painel de influencer com parcerias, grupo de seguidores, e métricas. Painel de crítico com selo
azul safira e avaliações com peso diferenciado.

Fase 11 — Cardápio Interativo e Sub-aba Avaliações
Cardápio com categorias reordenáveis (drag & drop), fotos e thumbnails por item, preços, e subaba “Avaliações” na página do estabelecimento com cards visuais e filtro por item.

Fase 12 — Chat e Compartilhamento
Chat em grupos (140 chars), chat support 1:1, lista de transmissão business, compartilhar para
grupos + Web Share API, seguir users + chat 1:1 mútuo, e mensagens diretas.

Fase 13 — Ajustes Business Panel
Bares clicáveis, campos read-only com solicitação ao suporte via chat inline, renomeação de
“Meu Bar” para “Meus Locais”, e vinculação automática de suporte.

Fase 14 — Feed de Fotos (Galeria)
Componentes PhotoGrid (4:5) e PhotoExpanded (9:16), sistema de curtidas e compartilhamento
de fotos, integração em MinhasAvaliacoes, PublicProfilePage e AvalyarinReviews.

Fase 15 — Ajustes Galeria + Logo
Logo 1:1 na página do estab, fallback para logo quando avaliação não tem foto, endpoints de
upload de logo (500x500) e capa (1200x800), e correção do bug “O” isolado.

Fase 16 — Detecção de Duplicidade
Tabela duplicate_alerts, detecção automática por telefone/endereço, sinalização manual pelo
suporte, revisão pelo admin, e validação de complemento obrigatório para multi-inquilinos.
